# Deployment Flow Reference

This document explains how this repository is intended to publish the HAT site through AWS Amplify.

## Short Answer

The live site should not be deployed by uploading the local `dist/` directory from this checkout.

Instead, deployment works like this:

1. Hugo source files in this repository are edited locally.
2. Changes are committed and pushed to the GitHub remote.
3. AWS Amplify detects the push on the connected branch.
4. Amplify installs npm dependencies and runs `npm run build` from the repository root.
5. Amplify publishes the generated files from `dist/`.

So `dist/` is a build output directory, not the authoritative deployment source.

## Deployment Source of Truth

The source of truth is the site source in this repository, primarily:

- `site/content/`
- `site/layouts/`
- `site/static/`
- `site/archetypes/`
- `src/`
- `webpack.*.js`

When those files change and the changes are pushed, AWS Amplify rebuilds the site.

## Why `dist/` Is Not The Source of Truth

This repository builds the deployable site with:

- webpack for frontend assets
- Hugo for site generation

The output is written into `dist/` by the existing npm build scripts. That directory should be treated as generated output.

## AWS Amplify Build Behavior

Amplify is expected to use this build flow:

```yaml
version: 1
frontend:
  phases:
    preBuild:
      commands:
        - npm ci
    build:
      commands:
        - npm run build
  artifacts:
    baseDirectory: dist
    files:
      - '**/*'
```

This means:

- Amplify starts from the repository root
- it installs the repo's npm dependencies
- it runs the same build command used locally
- it publishes the generated files from `dist/`

## Practical Publishing Flow

For day-to-day work, the intended flow is:

1. Edit content or templates in the repository.
2. Preview locally if needed.
3. Commit and push changes to GitHub.
4. Let AWS Amplify rebuild and deploy automatically.

The local CMS in `main.py` follows this same model. It edits source files; AWS Amplify handles the hosted rebuild.

## What To Edit

If you want to change the live site, edit the source files, for example:

- `site/content/**/*.md`
- `site/layouts/`
- `site/static/`
- `site/config.toml`
- `src/`

Do not treat `dist/` as the place to make permanent changes. Any files there can be regenerated by the next build.
